# 12 · SJIS UI 한글화 (System ②) — ✅ 통합 완료

> 아래의 설계·확장 위험 기록은 구현 당시 진단 근거다. 현재 슬롯 `0x86` 확장과 SJIS UI
> 재삽입은 통합 빌드와 실기 검증을 통과했다.

시작 저장메뉴(docs/11)를 넘어 **비압축 SJIS 텍스트 전 영역**(레이서명·팀명·파츠명·카테고리·메뉴·버튼프롬프트·다이얼로그)을 한글화한다. 압축-글리프 대사(673·어드벤처, System ①)와 **별개 경로**.

## 아키텍처 (build_menu.py 검증 메커니즘 확장 → build_sjis.py)

1. **글로벌 한글 타일뱅크**: 173음절 2bpp(16B)=2768B → 자유 ROM `$C1:9A00`. DMA로 VRAM 타일 768~ 상주(자유 224타일에 전부).
2. **렌더러 훅** `$C1:9696`→`$C1:9843`(docs/11): 마커 상위바이트 0xFE → 확장타일(VRAM 768+low). 렌더러 $C1:965E 단일 퍼널이라 훅 1개가 전 화면 커버.
3. **변환표 확장**: SJIS **미정의 리드 `0x85`** 전용. 행-오프셋 배열 `$C1:D1C3+0x0A`(0x85&0x7F=5, ×2)를 새 189슬롯 블록 `$C1:D5CB`(rowbase 0x408)로 리다이렉트. `0x85,0x40+i` → 한글 타일 i. 정상 SJIS에 0x85 리드 없음 → 충돌 0.
4. **폰트로더 DMA 훅**: ✅ **해결(2026-07-18)**. 트레이스(`trace_garage_vram.lua`)로 확정 —
   폰트를 VRAM 타일256에 올리는 로더는 **3곳**: 저장메뉴 `$C0:6F43`(기존 훅) + 개러지 `$C1:307F`
   + 선택화면 `$C1:1EFB`. 개러지·선택은 자체 로더라 한글이 안 실렸음(화면전환 `$C0:AD73`가
   타일0~2048 덮은 뒤 폰트만 재로드). → 두 로더의 폰트 DMA 트리거 `STA $420B`를 **JSL $C19980**로
   대체(루틴이 폰트DMA 실행 후 한글DMA까지 발사·RTL). 타일256 로더만 대상이라 그래픽 화면 무손상.
   시그니처 스캔으로 20개 트리거 중 VRAM대상=타일256인 2곳만 선별.
5. **글리프 세로정렬**: 게임 $D9 폰트=바닥정렬(행7), 8pt 한글=상단정렬(행0~6) → `GLYPH_SHIFT_DOWN=1`로
   1px 내려 무손실 바닥정렬(실기 확인). 2px는 최하단 획 잘림.
5. **문자열 재작성**: 티어별. 크롬은 대부분 in-place(한글이 더 짧음). 인명/팀/파츠는 포인터 테이블 경유 → 길면 재배치+포인터 패치.

## 구현 당시 착수 순서

- **S1** 메커니즘(뱅크+훅+표) + 티어 A 크롬 → build → **인게임 QA로 폰트로더 커버리지 검증**(리스크 조기 해소).
- **S2** 티어 B~E 인명·머신·팀(포인터 테이블 재배치).
- **S3(협업)** System ③ 그래픽: 사용자 아트 → 도구 주입(build_gfx 계열 재사용).

- **Team Running / Formation 설정 메뉴 스프라이트 깨짐**(2026-07-19 사용자 발견 → **2026-07-19 해소 확인**):

  full `build_all.py` 재빌드본에서 재현되지 않음**(사용자 실기 확인). 실기 테스트 진행 중.
- **원인(미확정, 유력 가설)**: 직전 `out/wgp2_kr.smc`가 프리즈 격리 중 만든 **스테일/부분 디버그 빌드**
  (no-SJIS ROM·어드벤처 원복 ROM 등)로 덮여 있어 그 불완전 화면이 깨져 보였을 가능성이 큼
  (next-session.md 경고와 정합). 클린 full 빌드가 7단계 전부(SJIS·그래픽·포메이션 폰트로더) 정상 반영 → 해소.
  - ⚠️ 이번 어드벤처 변경($CA 압축글리프 재할당)은 포메이션 SJIS 경로($D9 시트·$C1:9A00 한글 타일뱅크·별개 스텝
    `build_sjis`)와 **독립적** → 이 변경이 직접 고쳤을 개연성은 낮음. **원인 단정하지 않음**.
- **재발 대비(원 가설 보존)**: 재발 시 Mesen 트레이스로 포메이션 화면 **OBJ VRAM DMA 소스·타일범위** +
  khook2(`$C1:307F`·`$C1:1EFB`, 한글 → VRAM 타일 800 DMA) 실행여부 캡처.
  1. **한글 VRAM 경합**: 포메이션이 khook2 로더를 쓰고 VRAM 800+를 스프라이트 타일로 쓰면 한글이 OBJ 덮어씀(BG무손상·OBJ만=증상 일치).
  2. **재배치 겹침**: SJIS/그래픽 재배치가 포메이션 스프라이트 소스와 겹쳤을 가능성.
  - 수정안: (a) 포메이션 화면 khook2 미적용(로더 구분), 또는 (b) 한글을 스프라이트 미사용 VRAM으로 재배치.

## ⚠️ 진단 단서: SJIS 리드바이트 0x86 확장 (2026-07-21) — 회귀 의심 1순위

> **이후 SJIS/개러지/메뉴/일부 화면에 "글자 깨짐·엉뚱한 문자·빈칸·행 밀림"이 새로 생기면 이 확장을 최우선으로 의심하라.** 옵션 부품 14종 번역을 위해 SJIS 한글 슬롯을 189→224로 늘리며 **두 번째 SJIS 리드바이트 0x86**을 도입했다(사용자 승인 A). 커밋 전 상태에서 이 단서를 기록.

### 무엇을 바꿨나 (정확한 지점)

| 항목 | 값 |
|---|---|
| 신규 리드바이트 | **0x86** (기존 0x85에 추가). 음절 인덱스 `i<189`→`(0x85,0x40+i)`, `i≥189`→`(0x86,0x40+(i-189))` |
| 2차 변환블록 | **`$C1:D7F3`** (block1 `$C1:D679`+189×2=D7F3 직후). `block2[j]=0xFE00\|(189+j)` |
| 행-오프셋 패치 | 변환표 `$C1:D1C3 + (0x86&0x7F)*2 = $01D1CF` → block2로 리다이렉트(원래 미정의 공유블록 $C1:D25C 가리킴) |
| 총 음절 | **200** (0x85: 189 + 0x86: 11), VRAM 타일 800..999. **하드상한 = VRAM 224** |
| 렌더러 훅 | **무변경** (`$C1:9843`). marker `0xFE`의 low가 곧 VRAM 타일 오프셋이라 0x86 블록만 추가로 동작 |
| 코드/데이터 | `scripts/build_sjis.py`(KLEAD2·SLOT1·VRAM_CAP·TBL_BLOCK2, sjis_bytes 분기, block2 셋업, decode_kr) + `sjis_ui.json` tierF_options 15종 |

### 안전 근거 (커밋 시점 실측)

- **0x86은 게임 SJIS 텍스트에서 리드바이트로 안 쓰임**: banks C0/C1/C3/D0 전역 스캔 → 짝수정렬(리드) 0x86 후보 **0건**. 발견된 0x86은 전부 **트레일**(예 `83 86`=ユ)·**포인터테이블 데이터**(홀수 offset). 행-오프셋상 0x84~0x87이 미정의 공유블록을 가리켜 0x85와 동급으로 미정의.
- 기존 SJIS 문자열 **바이트 동일**(무회귀): 음절 배정 순서 안정(MENUS→기존티어→tierF) → 기존 음절 인덱스 0~179 유지=0x85, 신규 옵션부품 음절만 189~199=0x86.
- `build_all` 충돌가드 통과(SJIS 5201B, 충돌 0), System ① round-trip 681 lossless. CRC **8F9324C7**.

### 증상 → 진단 (이 확장을 의심할 단서)

| 증상 | 유력 원인 | 확인법 |
|---|---|---|
| **특정 기존 화면**(옵션부품 아닌 곳)의 일본어/기존한글이 한 글자만 깨짐 | 스캔 못한 영역에서 **0x86이 실제 리드로 쓰였다** → row 0x86 리다이렉트가 그 문자 깨뜨림. (스캔은 C0/C1/C3/D0만; 전투UI 등 미커버 가능) | 깨진 문자 위치의 원본 SJIS 바이트 확인 — `86 xx`면 정답. 해당 리드만 예외처리 or 0x87 등 다른 미정의 리드로 이전 |
| **옵션부품**(뒤쪽 음절)만 깨짐/빈칸, 앞 음절은 정상 | 0x86 **디코드/변환 경로**를 게임이 0x85와 다르게 처리 | Mesen에서 렌더러 `$C1:965E`/훅 `$C1:9843` 트레이스, block2 `$C1:D7F3` 값·행오프셋 `$01D1CF` 확인 |
| 개러지에서 **0x85 음절(기존 한글)까지** 깨짐 | **c1_ui 재배치 성장이 block1/2 침범** — 패킹이 극히 빡빡: c1_ui reloc는 `$C1:D5CB`부터, block1 `$C1:D679`, block2 `$C1:D7F3`로 **여유 거의 0**. 673 대사를 늘리면 reloc가 D679 넘어 충돌 | `build_patch` 출력 `c1_ui 재배치 NNN/10805B`와 D679 비교. 충돌가드가 잡아야 하나 순서 의존 |
| 한글 타일 일부 깨짐/전 화면 이상 | **VRAM 224 초과**(현재 200, 여유 24). SJIS 코퍼스 더 늘리면 VRAM 800+음절이 타 용도 타일 침범 | `build_sjis` 음절수 출력 확인. 224 넘으면 VRAM 재RE 필요 |

### 롤백 (원상복구)

- `git revert`(커밋 후) 또는 `scripts/build_sjis.py`·`sjis_ui.json`(tierF_options 제거)를 이전 상태로 → `build_all` 재빌드하면 0x85 단일 189슬롯 체계로 복귀(옵션부품만 다시 일본어). 렌더러·타일뱅크·기존 티어는 불변이라 안전.

### 남은 리스크 / 확장 여지

- 스캔 미커버 뱅크(전투·특수화면 SJIS)가 있으면 위 1행 증상 가능 → 회귀 발견 시 **전 뱅크 0x86 리드 재스캔**부터.
- 슬롯 상한은 이제 **VRAM 224**가 병목(변환표는 0x86으로 여유). README의 남은 대형 SJIS(레이서/팀 잔여·기타)까지 다 넣으려면 VRAM 예산 재확인 필요.
