# 12 · SJIS UI 한글화 (System ②) — 설계·진행

시작 저장메뉴(docs/11)를 넘어 **비압축 SJIS 텍스트 전 영역**(레이서명·팀명·파츠명·카테고리·메뉴·버튼프롬프트·다이얼로그)을 한글화한다. 압축-글리프 대사(673·어드벤처, System ①)와 **별개 경로**.

| 시스템 | 렌더 | 폰트 | 상태 |

## 아키텍처 (build_menu.py 검증 메커니즘 확장 → build_sjis.py)

1. **글로벌 한글 타일뱅크**: 173음절 2bpp(16B)=2768B → 자유 ROM `$C1:9A00`. DMA로 VRAM 타일 768~ 상주(자유 224타일에 전부).
2. **렌더러 훅** `$C1:9696`→`$C1:9843`(docs/11): 마커 상위바이트 0xFE → 확장타일(VRAM 768+low). 렌더러 $C1:965E 단일 퍼널이라 훅 1개가 전 화면 커버.
3. **변환표 확장**: SJIS **미정의 리드 `0x85`** 전용. 행-오프셋 배열 `$C1:D1C3+0x0A`(0x85&0x7F=5, ×2)를 새 189슬롯 블록 `$C1:D5CB`(rowbase 0x408)로 리다이렉트. `0x85,0x40+i` → 한글 타일 i. 정상 SJIS에 0x85 리드 없음 → 충돌 0.
4. **폰트로더 DMA 훅**: ✅ **해결(2026-07-18)**. 트레이스(`trace_garage_vram.lua`)로 확정 —
   폰트를 VRAM 타일256에 올리는 로더는 **3곳**: 저장메뉴 `$C0:6F43`(기존 훅) + 개러지 `$C1:307F`
   + 선택화면 `$C1:1EFB`. 개러지·선택은 자체 로더라 한글이 안 실렸음(화면전환 `$C0:AD73`가
   타일0~2048 덮은 뒤 폰트만 재로드). → 두 로더의 폰트 DMA 트리거 `STA $420B`를 **JSL $C19980**로
   대체(루틴이 폰트DMA 실행 후 한글DMA까지 발사·RTL). 타일256 로더만 대상이라 그래픽 화면 무손상.
   시그니처 스캔으로 20개 트리거 중 VRAM대상=타일256인 2곳만 선별.
5. **글리프 세로정렬**: 게임 $D9 폰트=바닥정렬(행7), 8pt 한글=상단정렬(행0~6) → `GLYPH_SHIFT_DOWN=1`로
   1px 내려 무손실 바닥정렬(실기 확인). 2px는 최하단 획 잘림.
5. **문자열 재작성**: 티어별. 크롬은 대부분 in-place(한글이 더 짧음). 인명/팀/파츠는 포인터 테이블 경유 → 길면 재배치+포인터 패치.

## 착수 순서 (슬라이스)

- **S1** 메커니즘(뱅크+훅+표) + 티어 A 크롬 → build → **인게임 QA로 폰트로더 커버리지 검증**(리스크 조기 해소).
- **S2** 티어 B~E 인명·머신·팀(포인터 테이블 재배치).
- **S3(협업)** System ③ 그래픽: 사용자 아트 → 도구 주입(build_gfx 계열 재사용).

## 🐛 알려진 이슈 (고칠 부분)

- **Team Running / Formation 설정 메뉴 스프라이트 깨짐**(2026-07-19 사용자 발견): 타일맵(BG)은 정상,
  **스프라이트 뷰어에서 깨짐**. 후보 원인 두 가지, 확인 필요:
  1. **한글 VRAM 경합**: 개러지·선택 폰트로더 훅(khook2 @ `$C1:307F`·`$C1:1EFB`)이 한글을 **VRAM 타일 800**
     에 DMA. 포메이션 화면이 그 로더를 쓰고 **VRAM 800+를 스프라이트 타일로 사용**하면 한글이 OBJ를
     덮어씀(BG 무손상·OBJ만 손상 = 증상 일치).
  2. **재배치 위치 변경**(사용자 지적): SJIS/그래픽 작업으로 어떤 자유공간·타일 위치가 바뀌어 포메이션
     스프라이트 소스와 겹쳤을 가능성. 개러지 그래픽 타일 관련 재배치 이력 점검 필요.
  - 확인법: Mesen 트레이스로 포메이션 화면의 **OBJ VRAM DMA 소스·타일범위** + khook2 실행여부 캡처.
    수정안: (a) 포메이션 화면 khook2 미적용(로더 구분), 또는 (b) 한글을 스프라이트 미사용 VRAM으로 재배치.
