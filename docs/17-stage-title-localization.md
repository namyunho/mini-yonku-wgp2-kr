# 스테이지 1~10 제목 한글화 — 세이브 문자열과 챕터 인트로 그래픽

## 결론

화면이 비슷해 보여도 제목 경로는 둘이다.

1. **세이브 파일 선택 화면의 챕터명**은 ROM `$C0:8234~$C0:82C8`의
   별도 10개 문자열이다. `$C0:830F`의 첫 10개 16비트 포인터가 이를
   가리키며, 대사와 같은 `$CA` 전역 가변 길이 글리프 인코딩을 사용한다.
2. **실제 챕터 시작 인트로**는 문자열 렌더가 아니라 `$D9/$DA`의 챕터별
   LZSS 2bpp 그래픽 10개다. BG3 VRAM byte `$2000~$2FFF`에 올라가며
   하단 `STAGE n`까지 자원에 포함된다.

`assets/translations/stage_titles.json`을 정본으로 추가하고 `build_patch.py`가
번역문을 원래의 연속 149바이트 영역 안에 다시 패킹한 뒤 첫 10개 포인터를
갱신한다. 이는 **세이브 파일 선택 화면** 전용이다. stage 11 이후 데이터가
시작하는 `$C0:82C9`와 포인터 테이블의 나머지 항목은 변경하지 않는다.

인트로는 사용자 승인본
`assets/graphics/stage_titles/stage_titles_workshop_approved.png`의 192x64 영역을
`scripts/build_stage_intro_titles.py`가 원본 자원의 제목 전용 192타일에
인코딩한다. 원문 제목 타일맵은 화면 좌우로 2타일씩 더 넓으므로 이 32타일은
투명화한다. `STAGE n`을 포함한 나머지 32타일과 포인터는 보존한다.

## 챕터 인트로 그래픽 실측

Mesen2에서 stage 1 인트로를 캡처해 화면·VRAM·WRAM을 대조했다.

- 레이어: **BG3**, 2bpp
- character data: VRAM byte `$2000~$2FFF` (0x1000B, 256타일)
- tilemap: VRAM byte `$E000`, 화면 대비 vertical tile offset `+1`
- 제목 작업 영역: 화면 `x32,y80,w192,h64`
- 실제 8x8 셀 정렬: 화면 `x32,y79`부터 24x8타일
- 원문 title tilemap의 실제 가로 범위: 화면 `x16..239`, 28x8타일
- 승인 본문은 가운데 **192타일**, 좌우 여백은 **32타일**이며 여백을
  투명화한다. `STAGE n` 등 나머지 **32타일**은 독립 보존한다.

28타일의 자원 배치는 행마다 단순 연속이 아니다. 위 4행은 14+14타일,
아래 4행은 16+12타일로 갈라진다. 아래 4행을 14+14로 잘못 해석하면
분할점 뒤의 승인 그림이 오른쪽으로 2타일 밀리고, 오른쪽 끝에 원문/승인
조각이 남는다. Mesen 캡처 `wgp2_kr_019.png`로 이 현상을 확인한 뒤 실제
BG3 tilemap 순서대로 수정했다.

원본 stage 1의 해제 자원 `$D9:B8EE`는 캡처 VRAM `$2000~$2FFF`와
4096/4096바이트 완전 일치했다. 같은 `$C3:C0C2` 표의 연속 10개 포인터가
아래 자원을 가리킨다.

| Stage | LZSS 자원 | raw | 원본 stream | 승인 stream |
|---:|---|---:|---:|---:|
| 1 | `$D9:B8EE` | 0x1000 | 1181B | 917B |
| 2 | `$D9:BD8D` | 0x1000 | 959B | 877B |
| 3 | `$D9:C14E` | 0x1000 | 1173B | 923B |
| 4 | `$D9:C5E5` | 0x1000 | 914B | 732B |
| 5 | `$D9:C979` | 0x1000 | 1029B | 798B |
| 6 | `$D9:CD80` | 0x1000 | 1207B | 885B |
| 7 | `$DA:0000` | 0x1000 | 924B | 765B |
| 8 | `$DA:039E` | 0x1000 | 990B | 810B |
| 9 | `$DA:077E` | 0x1000 | 777B | 656B |
| 10 | `$DA:0A89` | 0x1000 | 983B | 761B |

승인본은 전부 원래 stream 슬롯보다 작으므로 ROM 재배치나 확장이 없다.
각 슬롯은 2바이트 해제 길이 헤더(`0x1000`)와 LZSS stream만 갱신한다.

## 세이브 파일 제목 문자열의 IDA 역추적 근거

- 제목 포인터 테이블: `$C0:830F` (`PC 0x00830F`)
- 제목 원문 영역: `$C0:8234~$C0:82C8` (`PC 0x008234~0x0082C8`)
- 호출 함수: `$C0:75D3`
- 핵심 흐름:

```text
$C0:75FE  LDA $7EC2F5
            CMP #$FFFF
            BNE $C0:7615
$C0:7615  PEA #$00C0
            LDA $7EC2F7       ; stage index
            ASL
            TAX
$C0:761E  LDA.l $C0830F,X    ; title pointer
            PHA
            LDA #$0018
            JSL $C06634       ; title renderer wrapper
```

`$C0:6634`는 전달된 bank/pointer를 렌더 상태에 넣고 `$C0:6660`을 호출한다.
후자는 간접 long pointer에서 종료자 포함 스트림을 렌더 버퍼로 복사한다.
세이브 로드 경로 `$C1:8988`은 세이브 메타데이터에서 stage 값을
`$7EC2F5/$7EC2F7`로 복원한다. 이 결론은 세이브 파일 선택 화면 안에서는
유효하지만, 실제 챕터 인트로 그래픽 경로와는 공유되지 않는다.

## 전역 글리프 결합과 해결

제목 음절에 1바이트 슬롯 우선권을 주는 최초 방식은 기존 `$CA` 전역 글리프
인덱스를 바꾸었다. 그 결과 정적 대사 6개가 원본으로 남고 어드벤처에서 긴 런
48개가 생겼다. 이 방식은 폐기했다.

최종 방식은 기존 할당 순서(정적 대사 → 어드벤처)를 완전히 보존하고 제목에만
새로 필요한 음절을 마지막에 추가한다. 개별 원본 슬롯을 고집하지 않고 이미
역추적한 10개 포인터를 갱신하여 149바이트 전체를 공동 사용한다. 최종 제목은
136/149바이트이며 기존 글리프 매핑과 대사 압축 결과가 기준선과 동일하다.

2026-07-23 결과:

- 기존 정적 추출 681개: raw==ROM, encode(parse)==raw PASS
- 스테이지 제목 원문 10개: raw/포인터/재인코딩 PASS
- 스테이지 제목 한글 출력 역검증: 10/10, 136/149바이트
- 정적 대사 삽입 역검증: 681/681, 미커버 초과 0
- 어드벤처: 235씬/1,782메시지, 역검증 235/235, 긴 런 0
- 통합 `build_all.py`: SJIS 충돌 0, 2MB 유지, CRC32 `CF995A43`, MD5 `5a488bad3c6f42fcc4b52d31899dfafa`
- 최종 통합 ROM을 다시 읽은 제목 역검증: 10/10
- 챕터 인트로 승인 그래픽: 10/10 LZSS 왕복, 좌우 원문 여백 32타일 제거,
  `STAGE n` 포함 나머지 32타일 보존
- stage 1 원본 인트로 캡처와 원본 LZSS 해제 결과: 4096/4096B 일치

세이브 선택 문자열·포인터·글리프와 인트로 그래픽은 각각 최종 통합 ROM에서
자동 검증한다. 승인 인트로 10개는 Mesen2에서 챕터별 화면 위치와 그림자를
추가 실기 확인한다.
