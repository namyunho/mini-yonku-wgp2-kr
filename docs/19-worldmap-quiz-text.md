# 19 · 월드맵 퀴즈/정보 DB 한글화 — ✅ 번역·재삽입 완료

> 주소·렌더·재삽입 구현 정본. 70문항의 실제 문항·선택지·정답은 자동 생성 문서
> [`23-worldmap-quiz-audit.md`](23-worldmap-quiz-audit.md)를 사용한다.

## 결론

월드맵의 「무엇을 알고 싶나요？」 뒤에 표시되는 퀴즈를 전수 재검수했다. 최초 작업의
350개 질문/선택지 DB뿐 아니라, 그 밖에 있던 시작 안내·정답·오답 7문자열과 상단 상태줄까지
같은 회귀 게이트에 포함했다.

| 항목 | 결과 |
|---|---|
| 포인터 테이블 | `$C6:A08D`, 16-bit little-endian in-bank 포인터 **350개** |
| 원문 영역 | `$C6:A349..$C6:AB10` (`$C6:AB11`부터 타이틀 그래픽) |
| 구조 | 질문 1 + 선택지 4 = 5문자열/문항 |
| 문항 | 산수 40(표기 보존) + 정보 30(한글 번역) = **70문항** |
| 고정 UI | `$C6:A017..A07A` 시작 2 + 정답 3 + 오답 2 = **7문자열** |
| 상태줄 | `$C0:8D8A` 직접 타일 프로그램, 동적 문항 수·시간 칸 보존 |
| 재배치 | `$C6:E200..$C6:EA4F`, **2128/7680B** |
| 검증 | 원문 round-trip 350/350 + 7/7, 포인터·역디코드 350/350 + 7/7 |
| 정답 | `$C0:8E6D/$C0:8F39` 판정상 전 70문항의 원본 선택지 1 |
| 실기 | 시작·상태줄·정답/오답·긴 문항 한글 표시 확인 |

프롬프트의 “정보 35+”는 정적 경계를 잡기 전 추정치였다. 실제 정보 문항은 정확히 30개이며,
앞쪽에 별도 산수 문항 40개가 있어 DB 전체는 70문항이다.

## 오탐 정정: `$C6:8Bxx`는 독립 월드맵 문자열이 아니다

JP 글리프 바이트 검색으로 `$C6:8Bxx`에서 `やめて`·`コース` 등이 보이지만 이 구간은
**어드벤처 압축 딕셔너리 `$C6:7C73..$C6:9C56`** 안이다(docs/08). `$C0:3A4D`가
딕셔너리 조각을 읽어 압축 씬을 복원한다. 0x00 경계로 디코드하면 한 문자열 안에 무관한 일본어
조각과 바이너리가 섞이는 것이 오탐 신호다.

- 이 구간을 문자열처럼 직접 덮어쓰면 다수 어드벤처 씬의 압축 해제가 손상된다.
- `$C7:8Cxx/$C7:9Fxx` 검색 결과는 이미 `dialogue.json`의 681 정적 대사에 포함돼 한글화됐다.
- `$C7:A01D..A1AC`의 장소명 비슷한 문자열은 BGM/사운드 라벨 테이블이며 이번 월드맵 퀴즈
  런타임 경로와 무관하다.

따라서 질문/선택지 DB는 `$C6:A349..AB10` 하나다. 다만 2026-07-23 실기에서 DB 밖의
고정 메시지가 드러나 `$C6:A017..A07A`도 별도 포인터 클러스터로 추가했다.

## 포인터·렌더 경로 RE

IDA/idalib로 소비 루틴을 확인했다. 65816 M/X 문맥이 없는 함수 선두 일부는 IDA가 8비트 폭으로
오디코드하므로, 핵심 인덱스 계산은 원시 바이트와 교차 확인했다.

1. `$C0:8EAD` 상태 루틴이 선택된 문항/선택지 인덱스를 계산한다.
2. `$C0:8ED2`의 `LDA.l $C6A08D,X`가 350개 포인터 중 하나를 읽는다.
3. `PEA #$00C6` + 포인터를 스택에 두고 `JSL $C0:6634`를 호출한다.
4. `$C0:6634/$C0:6660`은 해당 주소부터 256B를 WRAM `$7E:A5E6`으로 복사한다.
5. `$C0:6699`가 `code >= $10 → glyph=code-$10`, 제어코드 점프테이블 방식으로 읽고
   글리프 페치 `$C0:6827`을 호출한다. 즉 기존 System①과 같은 `$CA` 글리프 시트를 사용한다.

문항 기준 포인터 오프셋은 `question_index * 10`(5포인터 × 2B)이며 선택지 순서 셔플이 여기에
`choice_index * 2`를 더한다. 문자열은 고정 슬롯이 아니라 포인터 테이블 경유이므로 전량 같은 뱅크
자유공간으로 재배치하는 것이 가장 안전하다.

## 2차 실기에서 발견한 DB 밖 UI

스크린샷의 첫 안내문과 오답 반응은 번역된 350개 DB가 아니었다. 퀴즈 종류별 데이터 구조체가
24-bit 포인터로 직접 참조한다.

| 용도 | 원문 주소 | 포인터 필드 |
|---|---|---|
| 산수 시작 | `$C6:A017` | `$C6:9F92`, `$C6:9FD0` |
| 산수 정답 | `$C6:A035`, `$C6:A040` | `$C6:9F95`, `$C6:9FD3` |
| 산수 오답 | `$C6:A044` | `$C6:9F98`, `$C6:9FD6` |
| 정보 시작 | `$C6:A04D` | `$C6:A00E` |
| 정보 정답/오답 | `$C6:A06C`, `$C6:A073` | `$C6:A011`, `$C6:A014` |

`scripts/extract_worldmap_text.py`가 7문자열의 원본 바이트·종료자·24-bit 포인터를 검증하고,
`scripts/build_worldmap.py`가 350개 DB 뒤에 함께 재배치한다.

상단 `あと{remaining}もん　のこり{time}`는 문자열이 아니라 `$C0:8D8A`의 직접 타일
프로그램이다. 월드맵 문맥 폰트가 가나 타일을 한글로 재사용하면서 깨져 보였다.
`scripts/build_menu4_reclean.py`에서 동적 숫자 칸과 소수점·버튼 타일은 그대로 두고 고정 가나만
지운 뒤, 이미 검증된 소형폰트 타일 `시($2F)`, 상태줄 원문을 지우며 비는
`간($01)`, 조사된 월드맵·튜토리얼 직접 프로그램에서 사용하지 않는
`문($1B)`·`제($1E)`로 `문제 {remaining}　시간 {time}`를 표시한다.

### `$1B/$1E` 타일 안전성 재검증

사용자 지적에 따라 “현재 번역 레코드에 안 보인다”보다 넓게 다시 감사했다.

- `$C0:8D5E`는 `$C0:8D8A`를 데이터 포인터로 쌓아 `$C0:1B4B`를 호출한다.
- `$C0:1B4B` 메인행은 `00`만 오버레이행 진입, `FF`만 빈 타일로 처리하고,
  나머지 값은 전부 `tile = byte + base`로 `$7E:0000` 타일맵 버퍼에 쓴다.
  따라서 `$1B/$1E`는 opcode·분기값·길이·종료자가 아닌 일반 타일 번호다.
- 원본 변환표와 2bpp 폰트에서 `$1B=ひ`, `$1E=ほ`인 실제 16바이트 글리프
  블록임을 확인했다.
- 지원 원본 전체에서 `JSL $C0:1B4B`는 정확히 10곳이다. 월드 폰트 문맥은
  상태줄·튜토리얼·월드맵 X 메뉴·세팅 메뉴 4곳, 나머지 6곳은 저장 화면의
  별도 폰트 문맥이다.
- 월드 폰트 문맥의 승인 직접 레코드 원문·제어 바이트에는 `$1B/$1E` 참조가
  0개다. 구조화된 SJIS UI 원문에도 `ひ/ほ`가 없다. `$C0:71E2`의
  `ほぞん`은 저장 화면 전용 `$C0:6F4B` 로더를 사용하므로 월드 폰트와 분리된다.
- 최종 `$C7:E000` 해제본을 원본 `$D9:0000` 해제본과 16바이트 타일 단위로
  대조한 결과 승인 87타일만 변경되고 승인 밖 변경은 0개다.
- 직전 `회/시간` 구현과 대조하면 `문제` 후속 변경이 더한 폰트 차이는
  `$1B/$1E` 두 개의 16바이트 글리프 블록뿐이다.
- 로더의 DMA 길이는 원본과 같은 `0x1000B`이며 코드 훅·NMI·추가 DMA는 없다.

`scripts/test_small_font_tile_safety.py`가 렌더러 시그니처, 10개 호출자 분모,
문맥별 참조, 원본 글리프 대응, 로더·DMA, 최종 변경 타일 집합을 매 통합
빌드에서 검사한다. 이 구조에서 다른 화면이 같은 타일을 참조하는 회귀가 생기면
빌드가 실패한다. 픽셀 블록은 실행 코드가 아니므로 `$1B/$1E` 교체가 CPU
제어 흐름을 바꾸거나 크래시를 만드는 경로도 없다.

정답 판정은 `$C0:8E6D`가 `[1,2,3,4]`를 만든 뒤 셔플하고, `$C0:8F39`가 선택값을
`DEC/BEQ`로 검사한다. 그러므로 버튼 위치는 매번 달라도 원본 선택지 1이 항상 정답이다.
70문항·네 선택지·정답 전체 목록은 `docs/23-worldmap-quiz-audit.md`가 정본이다.

## 산출물과 빌드

- `assets/translations/worldmap_kr.json`: 정보 30문항 번역 원장.
- `scripts/extract_worldmap_text.py`: 원본 해시·포인터·경계·글리프·round-trip을 검증하고
  `assets/translations/worldmap_text.json` 350엔트리를 생성한다.
- `scripts/build_worldmap.py`: 350문자열을 재인코딩·재배치하고 포인터 350개를 갱신한다.
- `scripts/build_patch.py`: 월드맵 신규 글리프를 기존 코퍼스 **뒤에만 append**한다.
- `scripts/build_all.py`: 어드벤처 뒤, SJIS 앞에 월드맵 빌더를 실행한다.

### 기존 글리프 할당 보존

월드맵 번역 빈도를 기존 681/어드벤처 빈도에 섞으면 1바이트 경계가 흔들려 긴 런 회귀가 생긴다.
그래서 기존 893개 한글 인덱스를 먼저 확정한 뒤 월드맵 전용 신규 8음절만 뒤에 추가했다.
산수 표기에 필요하지만 기존 패치가 보존하지 않은 `9`·`÷`·`＝`는 원본 `$CA` 타일을 새 슬롯 3개로
복제했다.

- 기존 `char2idx` 변경: **0개**
- 월드맵 신규 한글: `늬·징·썹·북·낭·켄·뀐·민` 8개
- 원본 기호 복제: `＝·9·÷` 3개
- 전체 한글: 901 / 자유 글리프 슬롯 969

- 전각 문장부호를 사용하고 원문 `{nl}/{end}` 시퀀스를 그대로 보존했다.
- 2차 실기 스크린샷에서 본문 시작 X=68, 오른쪽 테두리 직전 X=246을 확인했다. 단순 문자 수
  16칸 가드를 폐기하고 최종 ROM의 `$CA:9137` VWF 폭 테이블로 계산한다.
- 질문·선택지는 한 줄 **156px 이하**(여유 22px), 시작/정답/오답은 **178px 이하**이며,
  질문은 최대 2줄·선택지는 최대 1줄이다.
- 개별 문자열은 종료자 포함 **256B 미만**으로 유지한다.
- `glyph_table.tsv` 라벨/원문 결락 3곳은 raw를 바꾸지 않고 검수용 `jp`만 교정했다:
  `開会武→開会式`, `著て→着て`, `石　→石像`.
- 완역과 실제 표시가 다른 DB 문자열 9개·시작 안내 2개·상태줄은 원문/완역/표시문을 분리해
  `worldmap_kr.json`, `worldmap_text.json`, `docs/23-worldmap-quiz-audit.md`에 모두 보존한다.

## 실기 회귀 체크리스트

새로 생성한 `out/wgp2_kr.smc`로 다음을 확인한다.

1. 월드맵의 「무엇을 알고 싶나요？」 진입.
2. 시작 안내문과 상단 `문제 {남은 수}　시간 {시간}`이 깨지지 않는지.
3. 정답·오답 반응이 모두 한글인지.
4. 정보 문항 43·53·55·58·61에서 오른쪽 테두리 침범이 없는지.
5. 산수 문제에서 특히 `9`·`÷`·`＝`가 정상 표시되는지.
6. 월드맵을 나갔다 다시 들어와도 NPC/장소 번역·프리즈·리셋 문제가 없는지.

이 체크리스트는 후속 수정 시 같은 회귀가 재발하지 않는지 확인하는 용도로 유지한다.
