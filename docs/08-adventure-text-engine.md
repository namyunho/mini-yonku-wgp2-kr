## ✅ 2026-07-16 ④재삽입 PoC — **실기 한글 렌더 확인** (씬 0xC0 오프닝)

**결론: 재삽입 경로가 전부 열렸다.** 압축기 자작 → 재압축 → 자유공간 재배치 → **표 엔트리 3바이트 패치**로 끝.

### 압축기 (`adv_codec.compress` / `compress_scene`) — 게임 인코더 동급

- 코덱 역연산: 플래그바이트(LSB first) 8항목 / 비트0=리터럴(1B) / 비트1=딕참조(2B LE word=(idx<<3)|(len-1)).
  참조는 2B로 len B를 내므로 **len>=3 일 때만 이득** → 그리디 최장매치(3바이트 프리픽스 인덱스).
- **딕셔너리 = `$C6:7C73`, 길이 8164**(=씬테이블 `$C6:9C57` 직전에서 정확히 끝남. 인덱스 이론상한 8191과 정합).
  저부 0x00–0xFF = 아이덴티티 charmap 확인.
- **라운드트립 게이트: `decompress(compress(buf)) == buf` 250/250 PASS.**
- **크기: 원본(게임 인코더) 대비 총 −816 B**(동일 51·더작음 151·더큼 48[+1~3B]) → **동급**.

### 씬 저장 레이아웃 (재배치 설계 근거)

- **갭 0인 씬 212/249, 갭<0인 씬 0** → 씬들이 표 순서대로 **빈틈없이 연속 저장**. (코덱·헤더 모델의 추가 검증)
  ⇒ **대부분의 씬은 여유 0 → in-place 확장 불가. 재배치 필수.**
- 어드벤처 압축 데이터 실측 총량 ≈ **82 KB**(표 스팬 158.5KB 중 76KB는 씬 사이 비-씬 데이터).

### 자유공간 (재배치 가능 총량)

- 뱅크 **$C4–$DF**(2MB ROM 실체 범위. 씬 bank = `delta+$C4`)에서 길이≥512 0xFF/0x00 런 = **196.3 KB**.
- 이미 사용중 제외: `$C6:AB11`(21.7KB 타이틀로고 chr)·`$C7:B49B`(19.3KB c7 대사) → **≈155 KB 가용** ≫ 90 KB 필요.
- 후보(대형): `$CA:9537`(27.3KB, 폰트시트·폭테이블 뒤)·`$C9:B926`(18.1KB)·`$DA:BEDC`(16.7KB)·`$CF:BF9F`(16.5KB)·`$CB:C37E`(15.5KB)·`$CC:C9E0`(13.9KB)·`$D9:D239`(11.7KB).
- **씬은 뱅크 경계를 넘어도 됨**(디코더 `INC $11; BNE; INC $13`. 원본도 `$C4:FF7B→$C5:005D` 로 넘어감).

## ✅ 2026-07-16 ②씬 소스 열거 — **씬 소스 테이블 `$C6:9C57` 발견**(기존 "위치 미상" 관문 해소)

기존 docs가 "씬 소스는 게임 로직이 문맥적으로 설정, `LDA #$5950` 같은 즉치 0건 → 위치 미상"이라
결론냈던 건 **게임이 주소가 아니라 씬 id 를 넘기기 때문**이었다. id → 주소는 표가 담당한다.

- **씬 소스 테이블 = `$C6:9C57`** (파일 `0x069C57`), **3바이트/엔트리 = {addr_lo, addr_hi, bank_delta}**,
  **소스 = (bank_delta + $C4) : addr**. **250 엔트리(id 0x00–0xF9, 750B, `$C6:9C57..9F45`)**.
  뱅크 분포 $C4:158 / $C5:78 / $C6:14, 뱅크 전환까지 단조증가(`id 9D $C4:FF7B → id 9E $C5:005D`,
  `id EB $C5:FF9C → id EC $C6:0095`).
- **이 표를 쓰는 곳 3군데** (전부 `id×3` 색인 → `LDA $C69C57,X` / `LDA $C69C59,X; AND #$00FF; ADC #$00C4`):
  - **`$C0:3D1F`** = 게임로직의 씬 로더(A=씬id) → `JSL $C03CA2`(로더 본체)
  - **cmd 0x11 `$C0:4498`** = **서브스크립트 호출**(복귀 프레임 저장 → 상태7) — 오퍼랜드 = 씬 id 워드
  - **cmd 0x10 `$C0:444E`** = **씬 점프**(tail jump, 복귀 없음) — 오퍼랜드 = 씬 id 워드
  ⇒ cmd 0x10/0x11 이 **씬 참조 그래프**를 이룬다(스토리 진행 구조).
- **로더 본체 `$C0:3CA2`**: DP `$0A`=addr·`$0C`=bank 를 `$9A4D/$9A4F`에 저장, 커서 `$9A47`=0,
  `$9B70`(연산스택)·`$9C72`(푸시백) 클리어, **상태0 push**, 버퍼 `$9A49/$9A4B`=`$7E:9DDE`,
  좌마진 `$9A66`=`$14`·펜X `$9A68`=`$14` → `PEA #$007E;PEA #$9DDE; push $0C; push $0A; LDA #$0000; JSL $C039D5`.

### ⚠️ 압축 소스에 **2바이트 출력길이 헤더** (기존 adv_codec 의 시작오프셋 오해 해소)

```
$C0:39D5  LDA $05; BNE $39EB      ; A(=호출자 전달 길이)가 0 이 아니면 그대로 사용
          LDA [$11]; STA $05      ; ★ A=0 이면 소스 첫 워드에서 출력길이를 읽음
          INC $11 ×2 (+$13 캐리)  ; ★★ 무조건 실행 → 스트림 시작은 **항상 addr+2**
```
로더 `$3CFB` 가 `LDA #$0000` 으로 A=0 을 넘기므로 **길이는 항상 헤더에서** 온다(타이틀 LZSS와 동일 패턴).
⇒ `adv_codec.decompress_scene(rom, bank, addr, dic)` = 헤더 읽고 addr+2 부터 디컴프.

**교차검증**: `$C5:594E` 헤더 = **0x05E9** = ①의 VM 워크가 독립적으로 구한 오프닝 스크립트 길이
(스크립트 끝 0x05E8 → 길이 0x05E9)와 **정확히 일치**. 기존 트레이스 실측 `$C5:5950`·`$C4:3429` 는
표에 **없던 게 아니라 헤더 2바이트만큼 어긋나 있던 것** — 각각 **표 id 0xC0**(`$C5:594E`)·**id 0x70**(`$C4:3427`).

### 명령표 정정 (②과정에서 발견)

| cmd | 기존(정적추론) | **정정** | 근거 |
|---|---|---|---|
| 0x10 | 0B | **2B (씬 id) = 씬 점프** | `$444E`: 표 조회 + `JSL $C03D4B`, 복귀프레임 없음 |
| 0x11 | 0B | **2B (씬 id) = 서브스크립트 호출** | `$4498`: 프레임에 '커서+2' 저장($9A68,X) → 복귀시 `$3F84` 복원 ⇒ 부모 기준 2B 소비 |
| 0x53 | 1B+COND | **1B 고정** | `$52F2`: `BCC $530A`→즉시 RTS, 캐리셋 경로도 추가 바이트 안 읽음(플래그 셋만) |
| 0x54 | — | 1B + (op≠0 → 워드 2B) = 1 or 3 | `$56A0`: 캐리셋시 `LDY $9A47; LDA [$01],Y; INY;INY` |

또한 워커의 **cmd 0x20 off-by-one 정정**: 커서는 명령바이트 다음(=오퍼랜드 위치)이므로
**새 커서 = p+1+오퍼랜드**, 텍스트 = p+3 부터. (오프닝은 전부 cmd 0x21 이라 이 경로가 미노출이었음.)

## ✅ 2026-07-12 코덱 완전 해독 (엔진 $C0:39E0 정밀 디스어셈블 + 오프라인 디코더 검증)

`$C0:39E0` 디코더를 **손으로 정밀 디스어셈블**(선형 디스어셈블러는 M/X 폭 토글·16비트 LDY로 오싱크 → 원바이트 덤프로 재구성)해 코덱을 완전 확정. **오프라인 파이썬 디코더 `scripts/adv_codec.py`가 $C5:5950부터 실제 어드벤처 오프닝을 정확히 복원**(검증됨):
> 第1回WGP終了から1ヵ月後… / 土屋博士「やあみんな{nl}ひさしぶりだね / 豪「ああ！それで博士　話ってなんだ？ / 烈「第2回ワールドグランプリの{nl}ことじゃないですか？ / 藤吉「メンバーは当然{nl}この五人でゲスな？ / …

### 확정 코덱 (플래그바이트 LZ + 딕셔너리)
소스 = `[$11-$13]` 롱포인터에서 바이트 단위로 읽는 **단일 인터리브 스트림**. 플래그 비트 버퍼 `$03`(16비트, 상위바이트 0xFF 센티넬)이 8비트 소진 시 소스에서 플래그 바이트 1개를 리필(**LSB first**):
- **플래그 비트 0 = 리터럴**: 소스 1바이트를 그대로 출력.
- **플래그 비트 1 = 딕셔너리 참조**: 소스 2바이트(**LE**) = word → `index = word>>3`, `len = (word&7)+1` → 딕셔너리 `$C6:7C73[index]`에서 **len 바이트**를 출력에 복사.
- `Y(출력수) == $05(목표 글리프수)` 될 때까지 반복.

**출력 바이트열 = 파서(1바이트 가변길이) 인코딩과 동일**: `0x00`=끝, `0x01-04`=2바이트 글리프 프리픽스(빅엔디언, glyph=(b<<8|next)−0x10), `0x05`=개행, `0x06/07`=제어(1인자), 그 외 glyph=byte−0x10. → **어드벤처도 673과 같은 글리프표·폰트시트 $CA를 공유**. 딕셔너리 저부(0x00-0xFF)는 **아이덴티티 charmap**(dict[i]=i, 단일글리프용), 그 위는 실제 어구 프래그먼트(土屋·校長·ファイター·豪·烈·たまみ·タモツ·森…).

### 손 디스어셈블 (핵심 루프, $C0:39FC=루프탑, M=0/X16)
```
39F7 STZ $03 ; 39F9 LDY #$0000
39FC LSR $03                        ; 루프탑
39FE LDA $03 ; BIT #$0100 ; BNE $3A12   ; 플래그버퍼 남았으면 리필 스킵
3A05 LDA [$11] ; INC ptr ; ORA #$FF00 ; STA $03   ; 리필: 새 플래그바이트|센티넬
3A12 LDA $03 ; BIT #$0001 ; BNE $3A2E   ; 비트0=1 → 딕셔너리 참조
3A19 SEP#20 ; LDA [$11] ; STA [$15],Y ; REP#20 ; INC ptr ; INY   ; (비트0=0) 리터럴 1B
3A28 CPY $05 ; BCC $39FC ; BRA $3A5B(끝)
3A2E LDA [$11] ; INC ptr×2           ; 참조: 소스 2바이트(word)
3A3C PHA ; AND #$0007 ; INC A ; STA $01 ; PLA ; LSR×3 ; TAX   ; len=(w&7)+1, X=w>>3
```

### 텍스트 코드 (표 `$C0:413F`) — ⚠️ 기존 docs 2건 오독 정정

| code | 핸들러 | 의미 | 인자 |
|---|---|---|---|
| 00 | `$40B7` | 텍스트런 종료 | 0 |
| 01/02/03 | `$40CF` | 2바이트 글리프 `(code<<8\|next)−0x10` (`SEP#20;XBA;REP#20;JSR $415F;ORA;SBC #$0010`) | 1 |
| **04** | `$40E5` → `$49B1` | **입력대기**(상태0E push) | **0** |
| 05 | `$40EA` → `$4A1C` | 개행(펜X=좌마진 `$9A66`→`$9A68`) | 0 |
| **06** | `$40EF` → `$4979` | **박스 클리어**(상태0C push) | **0** |
| 07 | `$40F4` | 제어 | 1 (`JSR $415F`) |
| 08 | `$4117` | 표 `$C0:3ACF`→`$9A5E`(속도?) | 1 |
| 09 | `$4135` | `ASL`→`$9A64` | 1 |
| 0A–0F | `$40AA` | no-op(미사용) | 0 |
| ≥0x10 | `$4046` | **글리프 = code−0x10**, 폭표 `$CA:9137`, 펜X 상한 `#$00F0`, 넘으면 `$4177`(unget+wrap) | — |

1. **`0x04`는 글리프 프리픽스가 아니다.** 프리픽스는 01/02/03 뿐. 기존 "제어 = 글리프 0x3F0+" 는
   `04`의 *다음 바이트*를 글리프로 오독한 것. `0x3F6`=**`04 06`**(대기+클리어=메시지끝),
   `0x3F5`=**`04 05`**(대기+개행), `0x3F0`=**`04 00`**(대기+런종료). 통계(×341/×339/×259)는 우연히 정합.
2. **`0x00`은 "씬 구분바이트"가 아니라 텍스트런 종료자**(그리고 명령계층에선 script end/return).

- 페치 `$C0:415F`: 푸시백버퍼(`$9C72` 카운트/`$9C74` 스택)가 비었으면 `LDA [$05]; INC $05`. 줄바꿈 unget(`$4177`)·매크로 전개용.
- cmd 0x00 핸들러 `$C0:3F57`: 상태0→씬종료(`$9A56`=0) / 상태7→서브스크립트 복귀(`JSL $C03D4B` 재디컴프 + 커서 복원) / 그 외 pop 반복.

### 명령 오퍼랜드 = 4가지 소비 메커니즘

`scripts/adv_vm_map.py`가 각 핸들러를 선형 디스어셈블(VM 전역 `REP #$30` → M=0/X=0 고정이라 정확)해 자동 추론:
1. **고정 n바이트** — `LDY $9A47 … INY×n … STY $9A47`
2. **표현식** — `JSR $5B4A` (서브옵 `01`=push.w 2B, `02`=push.b 1B, 나머지 대다수 = 스택연산자 0B)
3. **조건부** — `JSR $5AC6` (ROM 전역 2곳: **cmd 0x53·0x54**). `op≠0 → 캐리SET → 워드 2B 추가`(총 3B); `op==0`이면 런타임 `$7E:99A5` 의존
4. **push한 상태가 소비** — cmd 0x50→상태16(`$5268`: 0x80회 후 `ADC #$0010` = **16B 블록**), cmd 0x51→상태17(`$52B6`: 1B씩 읽다 0x00에 pop = **0x00종료 바이트리스트**). cmd 0x02→상태1·cmd 0x04→상태3 = **길이한정 블록**(`CMP $9A6C/$9A66,X; BCC → JSR $3F40`)이라 선형워크가 자연 처리.

2. **딕셔너리 `$C6:7C73` 전체 덤프** + 엔진 정밀 디스어셈블(--m/--x 폭 추적)로 **오프라인 Python 디코더** 구현, adv_io.txt로 검증.
3. **어드벤처 텍스트 전량 추출**(소스 뱅크 $C4/$C5+ 범위 확정 — 포인터 테이블/스크립트 진입 파악) + 분량 집계.
4. **재인코딩 가능성 검증**: 딕셔너리 재구성 or 압축 우회(비압축 재기록 + 엔진 우회 패치) 설계 → 재삽입 방식 결정.
5. 이후 번역·재삽입.

## 관련 트레이스 스크립트 (`scripts/lua/`)
- `trace_text.lua` — 파서 $C1:9554 훅(대사 주소·호출처). 어드벤처 미경유 확인용.
- `trace_c6.lua` — $C6 읽기 PC 훅(딕셔너리 접근 규명).
- `trace_advptr.lua` — 어드벤처 소스 포인터 $11-$13 캡처(소스 뱅크·범위).
- `trace_advout.lua` — 디코드 출력·소스 바이트 캡처(코덱 역공학용).
- 실행: `Mesen.exe "<원본ROM>" "<script.lua>"` (원본 ROM으로 — 일본어 정상 표시). 산출 `tmp/trace/*.txt`.
