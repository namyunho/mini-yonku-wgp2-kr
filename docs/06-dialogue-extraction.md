# 06 · 대사 추출 (Dialogue Extraction) — ✅ 다중 블록 추출·완전성 닫힘(→ docs/07)

> **완전성 확정(docs/07)**: 파서 `$C1:9554` 호출처 7곳 전수 역추적으로 **정적 텍스트 뱅크 = $C7/$D0/$C1 셋**임을 권위 있게 닫음(+$7E WRAM 조립버퍼, $00 클리어). 전 블록 포인터 카탈로그(`assets/translations/pointer_catalog.json`) 확보. **c1 정정**: 이전 `c1_form`(15, $CFAF)은 포메이션 클러스터 하나만 — 실제 c1은 대사 4클러스터 38개(setting_save $C868·machine_names $C981·garage_grid $CE53·formation $CFDC)가 바이너리 데이터로 분리. **확정 정적 대사 총 674**(c7 232·d0 404·c1 38). 아래 651·$CFAF 기술은 docs/07로 대체.

인코딩 확정(docs/04) 이후, 대사 텍스트 블록을 **종료자 워크로 전수 추출**했다. **처음엔 $C7 한 블록(232)만 봤으나, RPG성 게임에 232개는 과소** — ROM 전역 스캔으로 **스토리/배틀 대사가 뱅크 $D0에 별도로** 있음을 발견, **총 651개**로 확장했다. 추출기 = `scripts/extract_dialogue.py`, 산출 = `assets/translations/dialogue.json` + `tmp/trace/extract_report.md`.

## 블록 탐색 방법 (ROM 전역)

232개가 전부가 아님을 확인하고, **글리프 표로 어떤 영역이든 디코드해 "일본어다움"을 판정**하는 전역 스캔을 썼다.

- **클린런 탐지**: 정렬된 워크가 미정의 op(0x08–0x0F)를 현재바이트로 만나지 않고 종료자 간격이 정상인 긴 연속 구간(>512B) = 텍스트 블록 후보. 코드·그래픽은 금방 정렬이 깨진다.
- **응집도 판정**: 후보 블록의 평균 글자수(≥5)·히라가나 비율(25–70%)·조사 밀도(≥10%)로 대사/테이블 분리. 단 **휴리스틱만으론 카나 테이블·SJIS를 완전 배제 못 해**, 최종 확정은 실측 디코드(사람 판독)로 했다.
- ⚠️ **완전성 한계**: 위 3블록은 실측 확정이지만, 소형 대사 블록·다른 텍스트 시스템이 더 있을 수 있다. 권위 있는 완전성은 각 대사 시스템의 포인터/opcode 역추적으로 확보한다(§다음 단계).

## 메시지 포인터 구조 (VM 역공학)

메시지 포인터는 정적 포인터 테이블이 아니라 **뱅크 $C3의 스레디드 코드 VM 명령 스트림**에 임베드돼 있다(동적 트레이스로 확정).

- **파서 호출 경로**: 파서 `$C1:9554` ← 호출자 `$C3:7899`가 `LDA $7EC58B,X`로 슬롯에서 텍스트 포인터를 읽어 Y로 넘김. `$7E:C58B`는 **범용 오브젝트 속성 배열**(X=슬롯). 대사 슬롯은 X=0x34였다.
- **슬롯 채우는 핸들러**: `$C3:CBEF`(및 `$C3:CBD4`) = "set-message" 명령. `LDA $0000,Y`(스트림에서 텍스트 포인터) → `STA $7EC58B,X`, 다음 워드 → `$7EC60B,X`(param). Y=명령 스트림 위치.
- **VM 명령 형식**: 스트림은 **2바이트 핸들러 주소(LE) + 오퍼랜드** 나열. 디스패처 `$C3:CE44 PEA/JMP ($1347)`가 핸들러 호출. 예(`$C3:D267`): `EF CB`(핸들러 $CBEF) `FF 89`(텍스트 $C7:89FF) `64 00`(param $0064).
- **주의**: `$7EC58B,X`는 오브젝트 위치 누적기로도 쓰인다(`$C3:CE6B`가 매 프레임 delta 가산). 슬롯 값만 보면 대사/비대사 구분이 안 되므로, **포인터 추출은 opcode(`EF CB`/`D4 CB`) 기준**으로 한다(값 범위 필터는 거짓양성).

→ 트레이스 근거: 레이스 대사 $C7:89FF의 포인터는 f=1086에 `$C3:CBF9`가 `$C3:D263`에서 읽어 슬롯에 기록. `$C3:D263`엔 실제로 `FF 89`가 있다.

## 추출 방법 (`scripts/extract_dialogue.py`)

1. **포인터 스캔**: 뱅크 $C3에서 `EF CB`/`D4 CB` opcode 뒤 2바이트 LE = $C7 텍스트 포인터 수집(77개).
2. **블록 워크**: $C7:89E2부터 종료자 0x00로 메시지 분할. 인코딩 모델(docs/04)로 2바이트 글리프(0x01–04)·제어(0x07) 폭을 반영해 전진.
3. **블록 끝 자동 탐지**: 최대 포인터 이후로 워크하다 유효성(stray 0x08–0x0F 없음, 길이 정상)이 연속으로 무너지면 종료.
4. **디코드·raw 보존**: 각 메시지에 `raw_hex`(종료자 포함) + `text`(글리프 인덱스 `[XXX]`·제어 `{nl}{end}...`) + `referenced_by`(포인터 소스).
5. **교차검증**: 모든 포인터가 메시지 경계에 착지하는지(현재 이탈 0 = 모델 정합).

### JSON 스키마 (`assets/translations/dialogue_c7.json`)

`table_id`, `source`(블록·인코딩·opcode), `stats`, `entries[]`(entry_id, c7_addr, file_offset, raw_hex, n_bytes, n_glyphs, text, referenced_by, flags). 번역·재삽입 단계의 입력.

## 검증 상태

- ✅ **포인터↔경계 정합**: 77 포인터 전부 메시지 경계 착지(이탈 0). 종료자 모델·포인터 모델 상호 증명.
- ✅ **엔드투엔드 앵커**: $89FF 엔트리 = 검증된 15글리프(「栄光の…」, docs/04)와 정확 일치.
- ✅ **라운드트립 무손실 증명(전 코퍼스)**: `encode(parse(text)) == raw_hex == ROM` 을 **전 232 메시지(5665B)에서 통과**. 인코딩 모델(2바이트 프리픽스 경계·제어코드·미해독0)이 앵커 1건이 아니라 전량에서 바이트 단위 정확함을 증명. 도구 `scripts/test_roundtrip.py`(회귀 게이트), 인코더 `decode_script.py`의 `parse()`+`encode()`(canonical 가변길이).
- ✅ **글리프 인덱스→문자 표 완성**: `assets/translation_guide/glyph_table.tsv`(**1008 엔트리**, 0x000-0x3EF). 폰트 시트($CA:1137) 전사 — 카나 오십음도 배열 + 기호·라틴·한자(교육한자+인명한자 순) 육안 전사. **코퍼스 전 232 메시지가 미매핑 0개로 완전한 일본어 디코드**(예: 「第1回世界グランプリ」「豪！マシンを信じろ！」「大逆転勝利」「鷹羽リョウくん」). 검증 앵커: 光=0x15C, 界=0x223, 第=0x25F, 勝=0x275, 栄=0x2ED, 烈=0x365, 豪=0x366, 優=0x372, 制=0x38B, 逆=0x3B1. 추출 JSON에 `text_jp`(일본어 원문) 필드 추가 → 번역 입력 준비 완료.
