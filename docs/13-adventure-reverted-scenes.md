# 13 · 어드벤처 원복·desync 복구 사건 원장 — 역사 기록

인게임 QA에서 번역이 VM을 깨뜨리는 씬(리셋·행·프리즈)을 발견해 한때 원본유지로 우회했다.
현재는 위치보존 빌드로 복구했으며, 이 문서는 원복 사유와 재적용 근거를 보존한다.
현재 구현 정본은 [`14-position-preserving-translation.md`](14-position-preserving-translation.md),
현재 메시지 수와 VM 완전성은 [`08-adventure-text-engine.md`](08-adventure-text-engine.md)다.

## 2026-07-23 엄격 VM 재감사 결과

과거 desync 원복 때 카탈로그에서 제외된 장면을 실제 재귀 표현식 경계로 다시 훑어
`0x7A/8B/99/A8/BE/C5/C6/E5/E6/F7` 10씬의 숨은 대사 **58런**을 복구했다.
특히 `0xA8` 14런과 `0xBE` 17런은 더 이상 앵커 재빌드가 아니라 원본 VM 소비 경계로
완주하며, `0xF7`은 `cmd0x5F`를 `0x00` 종료 목록으로 정정해 14런을 복구했다.

현재 기준은 **235씬·1,782메시지 전량 반영**, round-trip·역렌더 235/235, 긴 런 0이다.
원본 카탈로그 1,798런 가운데 나머지 16개는 대사가 아니라 과거 cmd0x20 over-read로
글자처럼 보였던 기능바이트이므로 빈 번역을 유지한다. 상세·분포는 docs/08,
회귀 게이트는 `scripts/test_adventure_completeness.py`가 정본이다.

## 당시 원본유지 씬 목록(현재 복구 완료)

### A. cmd 0x20 특수명령 (16씬) — ✅ 2026-07-20 위치보존 전환(리셋 해결)
메뉴 컨테이너·아이템 지급·레벨업 표시 등. cmd 0x20은 단순 텍스트런이 아니라 특수 명령이고,
op(길이/구조)를 재작성하면 VM이 깨진다(대화창 행·선택메뉴 미표시·씬 종료 후 리셋).

**✅ IDA 역공학 후 해결(2026-07-20, 직접 본문 복구 2026-07-22)**: cmd0x20 핸들러
`$C0:3FE6`의 2바이트 operand를 본문 길이에 맞춰 재작성한 것이 리셋 원인이었다. `build_adv.py`는
이제 헤더 3B(cmd+operand)를 원본 그대로 복사하고, 텍스트 본문만 원본 바이트 길이까지 패딩한다.

전수 감사 결과 `adventure_kr.json`의 번역 1,724개 중 기존 반영 1,715개, 숨은 cmd0x20 번역은
**9개/4씬(0x29·0x35·0x69·0x6F)**이었다. 6개는 그대로 들어갔고 3개는 원장에 완역을 보존한 채
축약했다. 빌드 게이트는 각 cmd0x20 operand, 디컴프 스크립트 길이, 전 런 `(at, cmd)`가 원본과
동일한지 검사한다. 결과: **231씬·1,724메시지 전량 역렌더 PASS, 긴 런 0**.
또한 번역 카탈로그 키와 실제 치환 키를 집합 대조해 하나라도 미반영이면 빌드를 실패시킨다.

| 씬 | 내용 |
|----|------|
| 0x29 | 이벤트 아이템 WGP 티켓 |
| 0x35 | 이벤트 아이템 안테나 |
| 0x64 | 파이터 코스엔트리(선택메뉴: 코스출전/그리드변경/그만두기) |
| 0x69 | **레벨이 올랐다(레벨업, 매 레이스 후)** |
| 0x6B | 휴일이라 잠겨 있다(잠김 메시지) |
| 0x6F | 아이템 입수 |
| 0xA7 | 레츠→카이조 반지 돌려주기 |
| 0xAC·0xAD·0xB3·0xB4·0xB5·0xB6·0xB7·0xB9·0xBA | 지로마루 야에짱 아크(특훈) |

### B. desync 앵커재빌드 (2씬) — 커밋 c782122
walk가 씬 구조를 못 잡아(desync) 앵커 기반 재빌드했으나 VM 유효성 미보장 → 원본유지.

| 씬 | 내용 |
|----|------|
| 0xA8 | 카이조 반지 분실 setup |
| 0xB2 | 지로마루 코스에서 반지 발견 |

### C. 조건분기 런타임 붕괴 씬 (프리즈) — build_adv `FREEZE_REVERT`
조건 분기(cmd 0x54)의 **런타임 제어흐름이 텍스트 번역(런 밀림)으로 깨져 프리즈**. 정적으로
원인 규명 실패(원본·빌드 둘 다 23런·desync 없음·구조 동일·제어코드 일치·명확한 offset 명령 없음).

| 씬 | 내용 | 상태 |
|----|------|------|
| 0xB0 | 료·카이 대화(카이 "큰소리치던 것에 비하면 대단할 건 없었네요") | **VM-붕괴 확정 후 복구 완료**. 0xB0 단독 원복으로 원인을 격리했고, 위치보존 프로토타입에서 긴 런 3개 축약+패딩과 실기 검증을 통과했다. |

**격리 과정(SSOT)**: no-SJIS ROM으로 SJIS khook2 배제 → 어드벤처 통째 원복=프리즈 소멸 → 창(0xAF/0xB0/0xB1)
바이섹션 → 0xB0 단독 원복=프리즈 소멸로 **0xB0 확정**. 씬 트레이서(`trace_scene_id.lua`)로 로드순서 확인.

⚠️ **프로액티브 전수탐지 불가 판정**: 조건분기(cmd 0x53/0x54) 씬이 **86개**(230 번역 중)인데 그중
0xB0만 깨진다. 정적으로 0xB0(깨짐)과 나머지 85개(정상)를 구별할 신호가 없음(offset 명령 hit비율 전부
낮음, cond 워드도 위치 불일치). → **깊은 cmd 0x54 스킵 메커니즘 RE 없이는 프로액티브 탐지 불가.**
현실적 방침: 확정 씬만 `FREEZE_REVERT`에 원본유지하고, 추가 프리즈/리셋은 **세이브+씬트레이서로 반응적**
확정 후 추가(각 ~15분). 근본 재번역은 cmd 0x54 스킵 RE 후 후속.

## 근본 재번역 방법 (완료/후속 기록)

1. **cmd 0x20 컨테이너**: ✅ 컨테이너 구조·op 보존 + 내부 텍스트만 같은 길이로 치환 완료.
2. **조건분기 씬**: cmd 0x54/점프의 **내부 offset을 텍스트 밀림 delta만큼 보정**하도록 build_adv 확장
   (어느 오퍼랜드가 씬내 offset인지 명령별 RE 필요). 또는 런 바이트길이 보존(패딩) 방식 검토.
3. **desync 씬**: 씬 구조 정밀 재파싱 후 텍스트런만 치환.

재번역 시 각 씬 build 후 **Mesen 실기 검증 필수**(round-trip 통과 ≠ VM 유효).
