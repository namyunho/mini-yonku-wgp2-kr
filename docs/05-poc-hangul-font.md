# 05 · PoC — 한글 폰트 교체·화면 표시 (Minimal Visibility Gate) ✅ 통과

본 구현 착수 전 게이트인 **최소 가시성 PoC**를 통과했다. 실제 게임 데이터 경로(스크립트 $C7 → 파서 $C1:9554 → 글리프 페치 $C0:6827 → 폰트 시트 $CA:1137)를 그대로 통해, 레이스 대사 자리에 **한글 "부천 레트로 흥해라"**를 표시했다. 번역·전량 폰트 생성 등 되돌리기 비싼 작업은 이 게이트 통과 후 진행한다.

- **결과 스크린샷**: `tmp/trace/poc_screen_1180.png` (Mesen2, frame 1180)
- **PoC ROM**: `out/poc_hangul.smc` (비커밋)
- **전제**: 대사 인코딩(`docs/04-dialogue-encoding.md`) + 본문 폰트 경로(`docs/03-font-survey.md`)

## 무엇을 증명했나

1. **폰트 시트 인코더(base03)가 정확하다** — 글리프 인덱스→시트 바이트 배치를 `render_font.py` 디코더의 정확한 역으로 구현. 정적 렌더(PNG)와 실기 화면 양쪽에서 한글이 또렷이 나옴.
2. **폰트 = 직접 바이트 패치로 충분** — 디컴프레서 불필요(비압축 확정과 정합). ROM 폰트 시트에 글리프를 직접 써 넣으면 게임이 그대로 렌더.
3. **글리프 인덱스 슬롯 교체만으로 화면 텍스트가 바뀐다** — 스크립트 바이트를 안 건드리고, 원문이 쓰는 글리프 인덱스의 폰트 슬롯을 한글로 덮으면 그 대사가 한글로 표시됨. (전량 한글화는 스크립트 재작성 + 글리프 대역 재배치가 필요 — font-strategy 단계.)
4. **VWF 폭 테이블로 자간·단어 공백을 제어** — 폭 테이블($CA:9137, 글리프 인덱스 1바이트 색인) 값으로 글자 advance를 조절. 공백(빈 글리프+폭6), 슬롯 숨김(폭0)까지 동작.

## 폰트 주입 방법 (핵심 사실)

### 폰트 시트 인코딩 (base03, `src/commands/poc_font.rs`)

- **시트 base**: `$CA:1137` = 파일 `0x0A1137`. 글리프당 16×16 1bpp = 32바이트.
- **글리프 인덱스 n → 시트 오프셋**: `base03(n) = 16*((n & ~7)*2 + (n & 7))`. 8글리프=256B 그룹, 상단블록 0~127 / 하단블록(+0x80) 128~255.
- **인코딩(디코더의 역)**: 각 행 R(0~15), `block = base(R<8) | base+0x80(R≥8)`, `r = R&7`. **좌 8px → `block+2r+1`, 우 8px → `block+2r`**, bit7=좌측 픽셀. 1bpp.
- **유효 글리프 인덱스**: `0x000..0x3FF`(1024). 시트 `0x0A1137`~`0x0A9137`, 이후 폭 테이블.
- **VWF 폭 테이블**: `$CA:9137` = 파일 `0x0A9137`. 글리프 인덱스로 1바이트 색인, 값 = advance 픽셀.

### 글리프 배치 규약 (16×16 셀)

- 원본 규약: **좌상단 정렬, 유효 잉크 ~11×11**, 우·하 ~5px는 VWF advance/행간 여백(`docs/03-font-survey.md` §5).
- 12×12 픽셀 한글은 `--xoff 1 --yoff 1`로 좌상단에 두면 셀에 자연스럽게 안착. advance는 폭 테이블 값(=13 권장)으로 별도 제어.

### 도구: `poc-font` 서브커맨드

```
cargo run -- poc-font \
  --rom "roms/<원본>.smc" --font "assets/fonts/body/x12y12pxMaruMinyaHangul.ttf" \
  --output "out/poc_hangul.smc" \
  --map "2ED=부:13,15C=천:13,028=:6,09E=레:13,0B5=트:13,088=로:13,02E=:6,054=흥:13,010=해:13,023=라:13,03B=:0,04E=:0,015=:0,051=:0,0D4=:0" \
  --px 12 --thr 128 --xoff 1 --yoff 1
```

- **`--map` 형식**: `글리프인덱스(hex)=문자[:폭]` 쉼표 구분. `문자` 생략(`idx=:6`)=빈 글리프(공백), `:0`=폭0(슬롯 숨김). 항목별 폭 > 전역 `--width` > 원본 유지.
- **`--px`**: 픽셀 폰트는 디자인 크기와 일치시킨다. MaruMinya는 unitsPerEm=1200·12px 디자인 → `--px 12`에서 1:1(스미어 없음, `--thr` 무관하게 선명).
- **`--thr`**: 아웃라인/AA 폰트(맑은 고딕 등)는 50%(≈128) 근처가 가장 깔끔. 픽셀 폰트는 아무 값이나.

## 사용할 폰트 (정리)

`assets/fonts/README.md`가 정본. 요지:

| 자산 | 내용 | 주입 | 라이선스 |
|------|------|------|----------|
| **`x12y12pxMaruMinyaHangul_2350.bin`** + `_glyphmap.json` | **완성형 2350자, 16×16 1bpp, 32B/글리프 사전 렌더** | **주력** — bin 글리프→픽셀 배열→`encode_glyph` | SIL OFL 1.1 |
| `x12y12pxMaruMinyaHangul.ttf` | 원본 아웃라인 | 대안/신규 글자 — `poc-font` fontdue 래스터 | SIL OFL 1.1 |
| 맑은 고딕 (`C:\Windows\Fonts\malgun.ttf`) | 시스템 아웃라인 | 초기 검증만 사용 | 시스템(**배포 불가**) |

- **주력 = 사전 렌더 `.bin`**: 게임 폰트 포맷(16×16 1bpp 32B)과 동일하게 이미 렌더돼 있어 즉시 주입 가능. 포맷은 **선형 행우선**(byte[2r]=좌8px, byte[2r+1]=우8px, MSB=좌) — 게임 시트의 블록분할·바이트순서와 다르므로 반드시 `encode_glyph`(base03) 경로로 재인코딩(README 참조).
- MaruMinya가 주력인 이유: 16×16 셀·유효 ~11×11에 12×12 픽셀 한글이 잘 맞고 OFL이라 배포 가능.
- 맑은 고딕은 배포 불가라 최종 미사용(초기 fontdue 검증에만 씀).
- **`.bin` 직접 주입 모드 구현됨**: `poc-font --bin ..._2350.bin --glyphmap ..._glyphmap.json --binyshift -2`. 문자별 우선순위 = 공백(빈 매핑) → `.bin`(glyphmap 조회) → TTF 폴백. `--font` 없이 `.bin`만으로 동작. `out/poc_hangul_bin.smc`가 이 경로로 생성한 "부천 레트로 흥해라"(전부 `[bin]` 소스, 정교한 16×16).
- **⚠️ `--binyshift -2` 필수(MaruMinya .bin)**: 이 `.bin`은 전 글리프가 잉크 행 2~12(상단 2px 여백)인데, 게임 원본은 상단 정렬(행 0~10, 하단 ~5px=줄간격). 시프트 없이 주입하면 **하단 ~2px가 잘린다**(줄간격 영역으로 밀림). `--binyshift -2`로 행0~10에 맞춰야 원본과 동일 정렬. (TTF 경로는 `--yoff`로 이미 상단 정렬되므로 무관.)

## 미해결 / 다음 단계 (본 구현)

이 PoC는 **글리프 슬롯 교체**만 증명했다. 전량 한글화는 아래가 필요하다.

0. **`poc-font`에 `.bin` 직접 주입 모드 추가**: `--bin assets/fonts/..._2350.bin --glyphmap ..._glyphmap.json` 입력 → 매핑의 문자를 glyphmap으로 인덱스 조회 → `bin[idx*32:]`를 선형 행우선으로 디코드 → 기존 `encode_glyph`로 주입. TTF/fontdue 경로와 공존(신규·특수 글자는 TTF).
1. **글리프 대역 설계(font-strategy)**: 한글 음절을 어느 글리프 인덱스 대역에 배치할지 결정. 후보: 원문이 안 쓰는 글리프 인덱스 회수 + 2바이트 프리픽스 공간(0x01–0x04 + lo, ~0x4EF). 완성형 수요는 번역 코퍼스 집계로 산정.
2. **전 글리프 인덱스→문자 표**: 폰트 시트 전사(추출 단계, `docs/04` 15앵커 확보). 원문 보존 글리프와 한글 신규 글리프 구분.
3. **스크립트 재작성 + 포인터 재배치**: 원문 대사를 한글 인코딩으로 재직렬화. ROM 메시지 포인터 테이블($7E:C58B 기록자 추적) 확정 필요.
4. **폭 테이블 일괄 생성**: 한글 글리프 advance(13~14px 권장) 일괄 기록.
5. **PoC 확장(선택)**: 다중 라인(개행 0x05)·다른 화면(메뉴 SJIS 경로)에서의 한글 표시 확인. 메뉴 폰트가 $CA 공유인지 별도인지 미확인(`docs/03` §3).
