# 03 · 폰트 경로 조사 (Font Path) — ✅ 본문 폰트 확정 (Mesen2 동적 트레이스)

Mesen2 Lua 동적 트레이싱으로 **본문(레이스/스토리 대사) 폰트 경로를 확정**했다. 이전 결론(“폰트 압축 추정”)은 **오류였다** — 폰트는 비압축으로 ROM 뱅크 **$CA**에 있으며, 정적 탐색이 이를 놓친 이유는 (1) 글리프 배열이 비선형(base03 매핑)이고 (2) 육안 검사 시 인터리브 디코더로 봐서 노이즈로 오판했기 때문이다.

## 결론 (핵심)

| 항목 | 값 |
|------|-----|
| 본문 폰트 위치 | **ROM 뱅크 $CA, base `$CA:1137` (파일 `0x0A1137`)** — 비압축 |
| 글리프 포맷 | **16×16, 1bpp, 32바이트/글리프** (2byte/행 × 16행). 8×8도 2bpp도 아님 — 실측으로 확정 |
| 글리프 비트 배치 | 상단 8행 = `topOff(n) = 0x0A1137 + 16*((n&~7)*2 + (n&7))` 블록(16B), 하단 8행 = `topOff(n)+0x80` 블록(16B). 각 행 = 16비트 리틀엔디언 워드(bit15=좌측 픽셀): **좌 8px = 둘째 바이트(off+2r+1), 우 8px = 첫째 바이트(off+2r)**. **1bpp 흑백** |
| 화면 표시 | **가변폭(VWF)**. 렌더러가 글리프를 픽셀 시프트해 실제 폭만 표시(한자 ~11px 폭·높이, 카나 ~7-9px). 16×16은 저장 셀 크기 |
| 글리프 배열 | 커스텀 순서: **숫자 0-9 → 히라가나 전체 → 카타카나 전체(탁점/반탁점 포함) → 라틴 A-Z → 문장부호·기호(♥ 포함) → 한자** (SJIS 순 아님) |
| 시트 범위 | `0x0A1137` ~ `~0x0A9537`(빈공간 시작), 8글리프=256B 그룹 → **≈1056 글리프** (카나+라틴+~900 한자) |
| 폰트 읽기 코드 | `$C0:686D LDA $CA1137,X` / `$C0:6875 LDA $CA11B7,X` (abs-long, DBR 무관) |
| 압축 | **없음.** 디컴프레서 경로 없이 ROM long-read 직접 → 폰트 교체 = 직접 바이트 패치 |

→ **프로젝트 난이도 대폭 하락**: 압축 대응(디컴프레서 역공학·재압축) 불필요. 본문 한글 글리프는 $CA 폰트 시트에 직접 주입한다.

## 렌더 경로 (동적 트레이스 실측)

레이스 대사 「栄光のゴールまであとわずかだ！」가 화면에 뜬 상태(frame 1200)를 기준으로 확정.

```
[스크립트 파서 $C1:95xx, DBR=$C7]           ← 대사 스크립트는 뱅크 $C7 (아래 주의)
   │  글자 코드 → $24/$26
   ├─ JSL $C0:6827  (글리프 페치)
   │     $C0:683B–6868 : 글자코드 $26/$24 → 폰트 오프셋 $03 계산 (base03 공식)
   │     $C0:686D LDA $CA1137,X(word) → DP $07  ┐ 8행 루프(행당 X+=2). 워드=상단 8행의 한 행(16px)
   │     $C0:6875 LDA $CA11B7,X(word) → DP $0B  ┘ +$80 블록=하단 8행. 각 워드=좌바이트+우바이트(16px 1bpp)
   │     $C0:687F/6885 JSR ($6A54,X)/($6A64,X)  : VWF 픽셀 시프트(가변폭 배치)
   │     $C0:6888 JSR $C0:68B2  (블리터)
   │        └ LDA $dp; ORA $A6E6,Y; STA $A6E6,Y ×16  : 시프트된 글리프를 WRAM 라인버퍼(2bpp VRAM 포맷)에 합성
   ├─ JSL $C0:6A84  (2차 처리)
   └─ 라인버퍼 클리어: $C0:6BC2 (STZ 루프, $7E:A6E6–$7E:AC00)
        │
        ▼ NMI/VBlank DMA (ch0, mode 1)
   WRAM $7E:A6E6  ──DMA──▶  VRAM word $4000 (byte $8000) = BG3 CHR (2bpp)
```

- 폰트는 ROM에 **1bpp**로 저장되지만, 블리터가 VWF 시프트 후 **BG3(2bpp) VRAM 포맷**으로 합성한다 (frame 1200: bgMode=1, BG3 chrAddr=word $4000, tilemapAddr=word $5C00).
- WRAM 라인버퍼($7E:A6E6~, 768B 단위)는 **타자기 효과용 합성 버퍼**일 뿐 — 폰트 원본이 아니다. 폰트 원본은 ROM $CA. (그래서 VRAM/WRAM DMA 소스가 전부 WRAM으로 보였으나, 그 WRAM은 ROM $CA에서 방금 복사된 글리프였다.)

## 재현 (도구)

- **동적 트레이스**: `scripts/lua/traces/dma_trace.lua`(모든 GP-DMA 로깅), `scripts/lua/traces/render_trace.lua`(라인버퍼 쓰기 PC·레지스터 집계), `scripts/lua/capture/dump_mem.lua`(WRAM/VRAM/CGRAM 덤프). Mesen2 실행: `Mesen.exe "<rom>" "<script.lua>"` (설정 `AllowIoOsAccess=true`, `ScriptTimeout=30` 필요 — 반영 완료). 스크립트가 Start 펄스로 타이틀→모드선택→레이스까지 진행 후 frame 1200에 산출.
- **폰트 렌더(정본 디코드)**: `scripts/render_font.py` — `off(n)=0x0A1137+16*((n&~7)*2+(n&7))`, 16B 인터리브 2bpp. 산출: `tmp/trace/img/font_full_*.png`(카나·라틴·한자 육안 확인 완료).
- 기존 `render-tiles`(선형 인터리브)는 base03 비선형 배열 때문에 이 폰트를 **선형으로 렌더하면 스크램블**된다 — 폰트 렌더엔 base03 매핑을 쓴 `render_font.py`를 사용.

## ⚠️ 미해결 / 다음 단계

1. **대사 인코딩(뱅크 $C7)**: 본문 대사는 **plain SJIS가 아니다**. 화면 대사 부분문자열(ゴール/あと/わずか)이 ROM·WRAM 어디에도 SJIS로 없음. 파서 $C1:95xx가 DBR=$C7로 스크립트를 읽어 **커스텀 글리프-인덱스 코드**($24/$26)로 변환한다. → 스크립트 위치·제어코드·글자코드→글리프 인덱스 매핑 역공학 필요(텍스트 추출 단계).
2. **글자코드 → 글리프 인덱스 매핑**: $C0:683B–6868의 산술(`$26/$24` → `$03`) 전개로 확정. base03 공식은 이 코드에서 추출.
3. **2번째 텍스트 시스템(메뉴/인명)**: 타이틀 메뉴·머신명·캐릭터명은 **plain SJIS**로 ROM $C0–$C1에 존재(02-text-survey 확인, `はじめから`=0x71B9 등). 이 경로의 **폰트가 $CA 공유인지 별도인지 미확인** — 별도 트레이스 필요(메뉴 화면 기준).
4. **글리프 2색(bp0/bp1) 정확 배치**: 코드는 bp1을 `$CA11B7`(+$80)에서 읽으나, 연속-16B 인터리브 디코드로도 글리프가 선명(형태는 bp0가 운반). 한글 주입 시 정확한 2색 레이아웃은 글리프 추출 단계에서 바이트 단위로 핀다운.
5. **글리프 앵커·여백(실측)**: 글리프는 16×16 셀의 **좌상단 정렬**, 유효 잉크는 대략 **11×11**(한자 기준, 상단 row0~10). 우측·하단 ~5px는 의도적 여백(우측=VWF가 실제 폭만 advance, 하단=줄 간격). → 한글 글리프도 **좌상단 정렬 + ~11×11 잉크 박스** 규약을 지켜 주입해야 원본 자간·행간과 맞는다.
6. **한글 인코딩 전략**: 16×16 셀(유효 ~11×11)은 8×8보다 한글 가독성 여유가 크다. VWF 폭 테이블(글자별 advance)의 위치·갱신 방식과, 한글을 어느 글리프 인덱스 대역에 배치할지를 `font-strategy` 단계에서 결정.

## 폐기된 이전 가설 (기록)

- ❌ “폰트 압축(ROM 15.3% 고엔트로피 블록이 폰트 후보)” — 오류. 본문 폰트는 $CA 비압축. 고엔트로피 블록은 코스/스프라이트 압축 그래픽으로 폰트 무관.
- ❌ “0x0A0000은 타일맵/희소 그래픽(폰트 아님)” — 오류. 바로 이 영역($0A1137~)이 본문 폰트 시트. 인터리브 디코더 + 선형 배열 가정으로 오판했다.
