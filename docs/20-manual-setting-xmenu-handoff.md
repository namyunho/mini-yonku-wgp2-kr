# 20 · 수동 세팅 X메뉴 한글화 인계

> 2026-07-22 Codex 독립 역추적 결과. 실패한 기능 패치는 제거했으며,
> 이 문서는 Claude 쪽 기존 파츠명·옵션명 한글화와 통합하기 위한 인계 자료다.

## 결론

수동 세팅에서 X로 여는 네 항목의 문자열 위치와 박스 구조는 확정됐다.
그러나 이 화면은 파츠명·옵션명·팀명·인물명과 같은 소형 글꼴 VRAM 페이지를
공유한다. 월드맵용 문맥 한글 폰트로 로더 전체를 교체하면 X메뉴는 정상 표시되지만,
배경의 기존 명칭들이 한글 글리프로 재매핑되어 깨진다.

따라서 다음 변경은 금지한다.

- 수동 세팅 진입 로더 `$C1:1EBB`를 월드맵 문맥 폰트 `$C7:E000`으로 변경
- 공유 글꼴의 기존 타일 번호를 X메뉴 전용 한글 음절로 덮어쓰기
- 화면이 보이는 동안 공유 VRAM 페이지 전체를 임시 교체

용어집·지도 커밋에는 수동 X메뉴 문자열과 로더 변경이 전혀 포함되지 않는다.

## 번역 대상

| 원문 | 목표 번역 |
|---|---|
| イージーセッティング | 쉬운 세팅 |
| パーツをはずす | 파츠 해제 |
| セッティングセーブ | 세팅 저장 |
| セッティングロード | 세팅 불러오기 |

필요한 한글 완성형은 14개다.

`쉬 운 세 팅 파 츠 해 제 저 장 불 러 오 기`

## 문자열과 박스 구조

전체 박스는 PC `0x01C6D4~0x01C75F`, SNES `$C1:C6D4~C75F`다.
직전 `$C1:C6D0`에는 `F1 81 49 00`이 있다.

각 행은 14바이트다.

```text
[E3][내부 11칸][E4][00]
```

- `E3/E4`: 좌·우 테두리
- `00`: 행 종단
- 본문 행 앞에는 탁점·반탁점만 둔 오버레이 행이 하나씩 존재
- 본문은 내부 첫 칸을 `FF` 여백으로 두고 최대 10칸을 사용
- 상단은 `E0 + E1×11 + E2 + 00`
- 하단은 `E5 + E6×11 + E7 + 00`

정확한 원본 행은 다음과 같다.

| 용도 | PC | 원본 바이트 |
|---|---:|---|
| 상단 | `0x01C6D4` | `E0 E1 E1 E1 E1 E1 E1 E1 E1 E1 E1 E1 E2 00` |
| 쉬운 세팅 오버레이 | `0x01C6E2` | `E3 FF FF FF 94 FF FF FF FF FF FF 94 E4 00` |
| 쉬운 세팅 본문 | `0x01C6F0` | `E3 FF 39 6F 43 6F 45 6E 4A 67 65 3F E4 00` |
| 파츠 해제 오버레이 | `0x01C6FE` | `E3 FF 95 FF FF FF FF 94 FF FF FF FF E4 00` |
| 파츠 해제 본문 | `0x01C70C` | `E3 FF 51 6F 49 2D 1A 0D 0D FF FF FF E4 00` |
| 세팅 저장 오버레이 | `0x01C71A` | `E3 FF FF FF FF FF FF 94 FF FF 94 FF E4 00` |
| 세팅 저장 본문 | `0x01C728` | `E3 FF 45 6E 4A 67 65 3F 45 6F 53 FF E4 00` |
| 세팅 불러오기 오버레이 | `0x01C736` | `E3 FF FF FF FF FF FF 94 FF FF 94 FF E4 00` |
| 세팅 불러오기 본문 | `0x01C744` | `E3 FF 45 6E 4A 67 65 3F 62 6F 4B FF E4 00` |
| 하단 | `0x01C752` | `E5 E6 E6 E6 E6 E6 E6 E6 E6 E6 E6 E6 E7 00` |

안전한 문자열 패치는 오버레이 행의 내부 11칸만 `FF`로 비우고,
본문 행은 `E3 FF + 번역 타일 + FF 패딩 + E4 00`으로 다시 조립해야 한다.
테두리 바이트를 패딩으로 덮으면 안 된다.

## 공유 글꼴 로더

수동 세팅 진입 중 다음 코드가 원본 소형 글꼴을 다시 읽는다.

| 항목 | 값 |
|---|---|
| 포인터 명령 | `$C1:1EBB` |
| 원본 바이트 | `F4 D9 00 F4 00 00` |
| 의미 | `PEA #$00D9; PEA #$0000` |
| 해제 호출 | `JSL $C0:0D52` at `$C1:1EC1` |
| 호출 복귀 | `$C1:1EC5` |
| 원본 자원 | `$D9:0000` |
| 해제 크기 | `0x1760` = 5,984B |
| DMA 원본 | `$7F:1000` |
| DMA 목적지 | VRAM word `$0800` |
| DMA 크기 | `0x1400` = 5,120B |

Mesen의 메뉴 열린 프레임은 PPU Mode 0이며 BG1/BG2/BG3의 CHR base가 모두 0이다.
라이브 VRAM `0x1000`의 앞 256타일이 원본 `$D9:0000`과 256/256 일치했다.
즉 X메뉴가 참조하는 글꼴은 월드맵에 남아 있던 `$4800` 페이지가 아니라,
수동 세팅 진입 때 이 두 번째 로더가 다시 올린 공유 페이지다.

같은 진입 루틴의 `$C1:1FA6`은 `F4 D9 00 F4 B9 12`로 `$D9:12B9`를
해제하고(`JSL $C0:0D52` at `$C1:1FAC`, 복귀 `$C1:1FB0`),
`$C1:20A7`에서 2,048B를 VRAM word `$3800`으로 전송한다. 이것은 글꼴이
아니라 BG3 타일맵 데이터다. 해결 대상을 `$D9:12B9`로 오인하면 안 된다.

## 실패 실험과 증상

### 1. 문자열만 한글 타일 번호로 변경

- 오버레이 행을 비우고 네 본문을 한글 타일 번호로 교체
- `$C1:1EBB` 로더는 원본 그대로 유지

결과: X메뉴에 일본어 가나처럼 보이는 잘못된 글리프가 표시됐다.
원인은 한글 타일 번호를 원본 `$D9:0000` 글꼴로 해석했기 때문이다.

### 2. 공유 로더도 월드맵 문맥 폰트로 변경

```text
$C1:1EBB
원본: F4 D9 00 F4 00 00
실험: F4 C7 00 F4 00 E0
```

결과: X메뉴 네 항목은 정상 한글로 표시됐다. 그러나 같은 페이지를 쓰는
파츠명·옵션명·팀명·인물명까지 월드맵용 한글 타일 대응으로 바뀌어 화면이
광범위하게 깨졌다. 사용자 실기 확인으로 재현됐다.

이 실험 출력은 CRC32 `B781D5E1`, MD5
`747c2d0602a632bf908518790cbd74d8`였으며 배포·커밋 대상이 아니다.

## Claude 통합 시 권장 방향

우선순위는 기존 파츠명·옵션명 한글화가 사용하는 글꼴·인코딩을 보존하는 것이다.

1. Claude 빌드의 수동 세팅 진입 직후 VRAM word `$0800` 페이지와
   `$C1:1EBB` 입력 자원을 먼저 덤프한다.
2. 기존 파츠명·옵션명·팀명·인물명이 사용하는 타일 번호 집합을 수집한다.
3. 공유 페이지에서 실제 미사용 타일 14개를 확보할 수 있다면, 그 슬롯에만
   위 14음절을 추가하고 X메뉴 본문을 그 번호로 인코딩한다.
4. 미사용 슬롯이 부족하면 기존 Claude의 CP932/System① 한글 렌더 경로로
   X메뉴 네 행만 보내는 전용 호출을 만드는 편이 안전하다.
5. 별도 DMA가 필요하다면 화면 전환/NMI 반복 주입이 아니라 X메뉴 생성 직전
   1회만 수행하고, 기존 이름들이 쓰지 않는 예약 슬롯만 갱신해야 한다.
6. 팝업 뒤의 파츠·옵션명이 동시에 보이므로 공유 페이지 전체 교체 방식은
   사용하지 않는다.

검증 장면은 최소한 다음을 포함해야 한다.

- 수동 세팅 X메뉴 네 행과 좌·우 테두리
- 파츠명과 옵션명 전체 순회
- 팀명·인물명 표시
- X메뉴 열기/닫기 반복
- 수동 세팅 이탈 후 월드맵 메뉴와 튜토리얼 재진입
- 화면 전환 중 보라색 노이즈 부재

## 현재 안전 상태

`scripts/build_menu4_reclean.py`의 용어집·지도 커밋에서는 다음을 제거했다.

- `$C1:C6D4~C75F` 수동 X메뉴 문자열 패치
- 월드 문맥 음절 72→80 확장
- `$C1:1EBB` 로더를 `$C7:E000`으로 돌리는 포인터 패치

따라서 현재 안전 빌드는 수동 세팅 화면과 기존 Claude 파츠·옵션 글꼴 경로를
수정하지 않는다.
